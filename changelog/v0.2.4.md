# v0.2.4 — 2026-07-18

The agent-ergonomics release: no new data sources — every change makes the
existing 14 tools easier for an LLM client to use correctly. Conventions
adopted from a study of the cyanheads MCP fleet.

## Changed

- **Error contracts.** Every refusal, hard error, and empty result now
  carries a machine-readable `structuredContent.reason` code
  (`not_covered`, `missing_required_filter`, `geocode_failed`, `no_match`,
  `missing_api_key`, `validation_failed`, `upstream_<kind>`,
  `unexpected_error`) plus an actionable `recovery` sentence. Legacy keys
  (`not_covered: true`, `error`) are retained, so existing clients keep
  working.
- Inconsistent error paths unified: geocode failures are now always
  `isError` with `reason: "geocode_failed"` (two tools previously returned
  them soft and text-only); FEMA no-zone / appraisal no-parcel / TEA empty
  results are soft `no_match` results with `count: 0` and recovery.
- **Every input field described.** The four missing `.describe()` calls
  (lat/lng on `dfw_fema_flood` / `dfw_appraisal`) added, with behavioral
  caveats — including the appraisal neighbor-parcel warning (geocodes land
  in the street right-of-way; verify the returned situs address). A unit
  test now enforces describe coverage.
- Cross-tool routing added to tool descriptions that lacked it
  (`dfw_district_lookup` as call-this-first, alerts vs flood zone,
  schools → district lookup), so routing survives clients that ignore
  server-level instructions.
- README gained a "How the tools chain" section (join keys + three
  documented chains).
- `dfw_nws_alerts` no longer silently truncates long alert instructions —
  truncation is marked with a pointer to the full JSON payload.

## Added

- `changelog/` per-version directory + assembled `CHANGELOG.md` (this
  file's convention), shipped in the npm tarball.
