# v0.2.3 — 2026-07-18

The Fort Worth + metro-city release: coverage beyond Dallas for permits,
code compliance, crime, and closures.

## Added

- `dfw_permits` and `dfw_code_cases` (new tools): Fort Worth (default),
  McKinney (`city:"mckinney"`), Arlington (`city:"arlington"`).
- `dfw_crime`: Fort Worth (`city:"fortworth"`) and Denton (`city:"denton"`,
  2019-11-06 → present) branches — explicit-only, never auto-detected from
  the address.
- `dfw_traffic` closures: Arlington ROW permits merged with Dallas, each
  result labeled with its `city`.
- New CKAN adapter (`lib/ckan.js`) for Denton — the first non-Socrata,
  non-ArcGIS upstream.

## Why it's shaped this way

- **Dallas is deliberately absent** from permits/code cases: every current
  Dallas permit feed is ~20 months stale and the code-case publication
  stalled 2025-01-31. Refusing beats serving stale data.
- **Irving is not wired at all**: its permit feeds, code-violation service,
  and police CSVs all froze around 2025-02-28 (the "…Present" service names
  lie); its events RSS is bot-blocked. Plano's record-level code datasets
  froze 2026-03. Both documented in `resources/datasets-index.md`.
- McKinney's live layers sit on an on-prem server, and its Energov permits
  layer has **no date field**: `address` is required for
  `city:"mckinney"`, ordering is by case number (roughly recent), and
  `since_date` is ignored with a note.
- Arlington's ROW layer publishes no per-record issued/created date (its
  `UpdatedInGIS` is a whole-table batch-sync stamp), so Arlington closures
  are ordered by a permit-ID year+sequence approximation — disclosed in
  output notes whenever an Arlington row appears.
- Denton's CKAN "Date/Time" is a text field that happens to sort
  chronologically; queried via `datastore_search_sql`.
